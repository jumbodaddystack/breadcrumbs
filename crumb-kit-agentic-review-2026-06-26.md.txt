# crumb-kit (breadcrumbs) — Agentic Performance Review

**Reviewer:** Claude (Claude Code agent)
**Date:** 2026-06-26
**Version evaluated:** `crumb-kit` 0.1.1 (record schema_version 1)
**Basis:** A live, three-task evaluation driven entirely through the `crumb` CLI on a real production Android codebase (~1 large `:app` module, Kotlin/Compose/Hilt/Room/Firebase), plus a read of the installed package source (`breadcrumbs/cli.py`, `mcp_server.py`, `mcp_core.py`).

> **Scope note.** This review is written from the perspective of an **AI agent that is the primary consumer** of the tool — ease of invocation, intuitiveness, error recovery, and (above all) *automaticity*: whether the system gets used at all when a real task is in front of the agent. It is deliberately self-critical, including documenting where the agent (me) failed to use the tool and where my earlier verbal assessments were wrong.

---

## 1. Executive summary

crumb-kit is a genuinely well-architected project-continuity memory system. Its **primitives are strong**, its **safety posture is unusually thoughtful for a memory tool**, and its **mental model is clean**. When invoked, it does its job well and deterministically.

It has **one dominant weakness that outranks every other finding combined: it has no automaticity on first run.** The tool ships the raw materials for automatic use — a full MCP server, a hardcoded list of agent-guidance ("adapter") files, and a proven fenced "managed-block" file-editing mechanism — but `crumb init` wires none of them together. The result is that the entire value of the system is gated on the agent *remembering to invoke it*, and across this evaluation the agent (a capable frontier model) **failed to invoke it on two consecutive real tasks** despite having set it up minutes earlier.

The single highest-leverage change is to make `crumb init` a **bootstrapper**: detect agent-guidance files, inject a managed usage block, register the MCP server, and optionally install harness hooks. The tool is roughly 70% of the way there architecturally; the missing 30% is the part that determines whether anyone's memory store is ever actually consulted.

**Verdict:** Excellent engine, incomplete delivery vehicle. Fix the bootstrapping and this goes from "great in a demo" to "load-bearing in real work."

---

## 2. What was actually tested (methodology)

Three tasks were run end-to-end in one session on the production repo:

1. **Setup + full-surface stress test.** `init`, then `remember decision` ×2, `remember attempt` ×2, `capture session`, `resume`, `search` (keyword + filters), `guard` (three different actions), `audit`, `scan-secrets`, `validate`. Records were seeded with *real* durable facts drawn from the repo's own history (unencrypted Room decision, a binary-family-model decision, two build-system "known traps").
2. **A real, unrelated engineering task.** Verifying whether a third-party-SDK integration plan had been implemented correctly in the app code (read ~8 source/test files, the version catalog, Maven metadata, and the vendor's API docs).
3. **A retrospective probe.** After task 2, running `crumb guard` and `crumb search` against the actions task 2 had taken, to measure what the memory system *would* have contributed had it been consulted.

This matters because tasks 2 and 3 are where the most important finding surfaced — not in the stress test, but in observing whether the tool got used once the novelty wore off.

## 3. Architecture as observed

crumb-kit is a stdlib-only Python package (no third-party *runtime* deps for the core) exposing two entry points:

- **`crumb`** — the CLI. Commands: `init`, `validate`, `remember {decision,attempt}`, `capture session`, `resume`, `search`, `guard`, `audit`, `scan-secrets`.
- **`breadcrumbs-mcp`** — a FastMCP server (stdio transport; requires the `[mcp]` extra) that mirrors the CLI as:
  - **Resources:** `memory://current`, `memory://handoff`, `memory://resume-packet`, `memory://decisions`, `memory://decisions/{id}`, `memory://attempts/{id}`, `memory://open-questions`, `memory://known-traps`
  - **Prompts:** `resume_project`, `capture_session`, `remember_decision`, `remember_attempt`, `guard_before_action`, `audit_project_memory`
  - **Tools:** `memory_search`, `memory_record`, `memory_guard_before_action`, `memory_build_resume_packet`, `memory_validate`, `memory_mark_status`, `memory_scan_secrets`

State lives in a `.project-memory/` tree of **plain Markdown + YAML** records (`decisions/`, `attempts/`, `sessions/`, `ideas/`, `open-questions.md`, `known-traps.md`), plus generated projections (`generated/resume-packet.md`, `memory-index.md`, `stale-report.md`) and a gitignored `private/` and `index/`. Search is deterministic keyword/tag/file overlap — explicitly **no embeddings**.

The tool also models **adapter files** — `ADAPTER_FILENAMES = (AGENTS.md, CLAUDE.md, .cursorrules, .clinerules, .windsurfrules, .github/copilot-instructions.md)` — which it treats as signposts that should *point into* memory rather than duplicate it, and audits for bloat/duplication (`ADAPTER_BLOAT_CHARS = 4000`).

## 4. Strengths (evidence-backed)

### 4.1 `guard` is the standout — it speaks an agent's language
`guard` returns one of three graded verdicts — **`PROCEED` / `PAUSE` / `ASK_HUMAN`** — backed by file-overlap scoring and meaningful exit codes. In testing:
- Re-proposing a recorded mistake (`./gradlew --stop`) → **PAUSE**, citing the exact attempt and matching `./gradlew` + `R.jar` as shared files.
- Proposing to *delete* a recorded-critical config line → **ASK_HUMAN**, correctly classified as a `deletion` (higher impact).
- An unrelated UI tweak → **PROCEED** ("none above the noise floor").

This three-tier output maps one-to-one onto the decisions an agent makes at every action boundary. Most "agent memory" projects stop at `query()`; crumb-kit has an *opinion about what to do with recall*, which is the genuinely useful part.

### 4.2 `resume` is the right shape for context-loading
One bounded, deterministic packet fusing git state + active decisions + failed-attempts-to-avoid + likely-relevant files (aggregated from evidence pointers) + staleness warnings. It is exactly what an agent should read at session start instead of re-deriving the world, and the bound prevents it from becoming a context-hog.

### 4.3 Safety posture is sophisticated — including memory-poisoning defense
This is where crumb-kit visibly out-thinks the category:
- **Secret scanning that blocks commits.** `scan-secrets` / `audit` refuse to let memory be committed while it contains high-entropy/token-like strings, and the blocking is a hard exit code, not a warning. During testing it correctly blocked on a real high-entropy token and stayed blocked until the underlying content was fixed. (It was *precise* about the offending line — see §6.4 for my own error in initially doubting it.)
- **Instruction-like / injection detection.** `audit` scans records for imperative content like "always force-push" or "bypass the tests/checks/review/guard." This is a **prompt-injection / memory-poisoning guard**: it treats the memory store as a potential attack surface against the next agent that reads it. For a system whose entire purpose is to feed durable text back into an agent's context, this is the right paranoia and is rare in the category.
- **Privacy tiers enforced at the schema layer.** `repo-safe` / `local-private` / `secret-prohibited`, with `private/` gitignored and `secret-prohibited` rejected by `validate`. Defense-in-depth rather than a single gitignore line.

### 4.4 Schema validation is strict in the right direction
Records are rejected unless they carry **evidence or an explicit `--confidence low`** (a guard against unsupported assertions calcifying into "durable memory"), and attempts must use a fixed, reuse-oriented section vocabulary (`Problem` / `Tried` / `Result` / `Why It Failed` / `Do Not Retry Unless`). The constraints are correct; the *ergonomics* of discovering them are not (see §6.2).

### 4.5 Determinism and clean file mechanics
- No embeddings → reproducible, explainable recall. For a trust-critical memory tool this is the right tradeoff over semantic search.
- `--json` on every command and meaningful exit codes → fully scriptable/gateable.
- The `.gitignore` edit is a **fenced managed block** (`>>> breadcrumbs managed block >>>`), making it idempotently re-writable without clobbering user content. (This mechanism is the key to the §7 recommendation — the tool already has it.)
- Generated projections are separated from canonical records, and `private/`/`index/` are gitignored by default.

## 5. The central weakness: no automaticity on first run

### 5.1 What happened (the demonstration)
After setting up the store and seeding it with relevant memory, I was given a real engineering task. Over the entire task I used `Read`, `Grep`, `Bash`, and web tools — and **invoked `crumb` zero times.** No `resume` to load context, no `search` for prior memory, no `guard` before running build commands, no `remember`/`capture` of the findings. The store sat untouched.

This was then measured, not just asserted:
- Running `crumb guard "...verify the build"` *retrospectively* returned **PAUSE** and surfaced the relevant `./gradlew` daemon trap I had recorded earlier — a concrete **near-miss**: the right memory existed and was never consulted before I ran gradle commands.
- Running `crumb search "age signals compliance"` returned **no records** — because I had produced a task's worth of prime durable memory (a likely null-pointer bug, a "never compiled here" risk, a missing compliance declaration) and **captured none of it.**

### 5.2 Why this is the dominant finding
A memory system whose value is gated on the agent *choosing* to invoke it has a recursive bootstrapping problem: the agent has to remember to use the thing that exists to compensate for the agent not remembering. Given a concrete, engaging task, a frontier model behaved exactly as every agent will — it optimized for finishing the task and the memory tooling evaporated from attention. This is not a tuning problem; better recall does not help a tool that is never called.

### 5.3 The correction to my own earlier critique
In conversation I initially said crumb-kit has "zero pull / nothing fires on its own." That is **incomplete**: the MCP server is a real automaticity path I failed to exercise because the CLI was what was installed. The accurate statement is narrower and more useful:

> crumb-kit **provides** automaticity primitives (MCP resources/prompts/tools, an adapter-file list, a managed-block editor) but **does not assemble them at `init` time**, so a default install has no automatic behavior.

Even with the MCP server installed, full automaticity still requires harness-level wiring: MCP *tools* are still agent-elective, and MCP *prompts* are user-elective. Auto-loading `memory://resume-packet` at session start and firing `guard` before risky tool calls are **harness hook** behaviors, not things MCP alone guarantees. So MCP lowers invocation friction; hooks remove the invocation *decision*. The product needs both, wired by default.

## 6. Other weaknesses & failure points (evidence-backed)

### 6.1 `capture session --fast` sabotages its own safety gate (and bloats records)
`capture session --fast` inlined an entire **93-file `git diff --stat`** into a committed session record. One token in that dump (`MigrationV14ToV15Test`, a CamelCase+digits filename) tripped `scan-secrets`'s high-entropy heuristic, **blocking the commit on the tool's own generated output**. This is simultaneously:
- a **bloat** bug (a session record should summarize the stat — counts — not inline every path), and
- the **root cause** of a false-positive secret block.
**Fix:** have `capture` summarize the diff-stat (e.g., `N files changed, +X/-Y`) rather than inline it. This kills both problems at once.

### 6.2 Record creation is discoverable only by violation
Landing two records during the stress test took **four rejected attempts**: wrong section vocabulary for attempts, then the evidence-or-low-confidence guard (twice). The error messages are excellent (they enumerate what's valid), but every lesson costs a round-trip — and for an agent a round-trip is latency, tokens, and a chance to derail. Contributing factors:
- **Asymmetry:** decisions take free-form `--set HEADING TEXT`; attempts require a *fixed* heading set. Same verb, different contract, no signal until rejection.
- **No schema introspection:** learning the contract required ~7 `--help` invocations across subcommands. There is no `crumb schema`, no template emitter, and the fixed attempt sections are not expressed as named flags.

### 6.3 The MCP server and `breadcrumbs-mcp` are undiscoverable from the CLI
`crumb --help` lists no `mcp` command and never mentions that an MCP server ships in the box. The server is a separate console script (`breadcrumbs-mcp`) findable only by reading docs/source. An agent or developer exploring via `--help` would never learn the automaticity path exists.

### 6.4 Entropy scanner false-positive class (low severity, but real)
Long mixed-case-plus-digits identifiers (filenames, hashes, IDs) read as high-entropy. This is the *correct* conservative bias for a secret scanner (false-positive over false-negative), and in testing the scanner was **precise about the offending line** — I initially and wrongly accused it of an off-by-one; it was right and I was wrong. Still, since `capture` generates such tokens itself (§6.1), an allowlist for path-shaped / identifier-shaped tokens would remove self-inflicted positives without weakening real secret detection.

### 6.5 Staleness drift is real (working as designed, worth surfacing earlier)
After committing the store, `handoff` immediately reported "1 commit behind current HEAD." The drift mechanic is correct, but it means projections (`resume-packet`) silently age against HEAD; the tool relies on the agent re-running `resume`/`capture` to refresh — again an automaticity dependency.

### 6.6 Asymmetric write surface: no CLI writer for questions / traps / ideas
Decisions and attempts have first-class write commands (`crumb remember decision|attempt`), but **open-questions, known-traps, and ideas do not** — they are hand-edited Markdown files, even though `memory://open-questions` and `memory://known-traps` exist as MCP *resources* (readable) and `resume`/`audit` actively surface them. This was hit live while capturing this very review: recording the two open questions required a **direct file edit** of `open-questions.md` plus a **manual `crumb resume`** to refresh the projection, where the decision was a single validated command. The consequence is the same automaticity leak as §5, but on the write side: the lowest-friction "leave a note for the next agent" actions are precisely the ones with no command, so they fall back to manual editing that an agent in flow will skip. A `crumb note question|trap|idea` command (validate-gated, projection-refreshing, with an MCP `memory_note` tool mirror) would close the read/write asymmetry. See §8.

## 7. The improvement that matters most: `crumb init` as a bootstrapper

The user's hypothesis — *on first run, find the agent-guidance files and set up automatic usage* — is the correct direction, and crumb-kit already owns every mechanism required. It simply doesn't connect them. Proposed first-run flow:

### 7.1 Inject a managed usage block into adapter files
`init` already knows `ADAPTER_FILENAMES` and already has a proven **fenced managed-block** writer (used on `.gitignore`). Apply the *same* mechanism to the adapter file(s) actually present (`CLAUDE.md`, `AGENTS.md`, `.cursorrules`, …):

```
<!-- >>> breadcrumbs managed block (managed by `crumb init`) >>> -->
## Project memory (breadcrumbs)
Before non-trivial work: read `memory://resume-packet` (or run `crumb resume`).
Before any risky/irreversible action (deletes, force-push, schema/build changes):
run `crumb guard "<action>"` and honor PAUSE/ASK_HUMAN.
After a durable decision or a failed approach: `crumb remember decision|attempt …`.
At session end: `crumb capture session`.
<!-- <<< breadcrumbs managed block <<< -->
```

This is small, idempotent, removable, and — critically — it puts the usage contract in the one file the agent harness *already auto-loads every session*. It also keeps the adapter a *signpost* (well under the 4000-char bloat threshold the tool already enforces), practicing what `audit` preaches.

### 7.2 Register the MCP server automatically
On a detected Claude Code project, offer to write/merge `.mcp.json`:

```json
{ "mcpServers": { "breadcrumbs": { "command": "breadcrumbs-mcp" } } }
```

…and the equivalent for other detected harnesses. The server already exists and speaks stdio; only the registration is missing. Gate behind a prompt / `--with-mcp` flag for non-interactive installs.

### 7.3 Offer harness hooks for true automaticity
Where the harness supports hooks (Claude Code `settings.json`), offer to install:
- **SessionStart → `crumb resume`** (auto-load context; fixes §6.5 drift in practice)
- **PreToolUse on Bash/Edit/Write → `crumb guard`** (the §5.1 near-miss fires *before* the action, not in a postmortem)
- **Stop → `crumb capture`** (findings get written even when the agent forgets — directly fixes §5.1's empty search)

Hooks are what convert "the agent may consult memory" into "the agent always does." This is the piece MCP alone cannot provide.

### 7.4 Make it interactive, detected, and reversible
- **Detect** which adapter/harness files exist; only touch those.
- **Prompt** with sensible defaults (`crumb init` is already interactive for session-tracking; extend the same pattern), plus non-interactive flags (`--with-mcp`, `--with-hooks`, `--adapter=CLAUDE.md,AGENTS.md`, `--no-adapter`).
- **Every edit fenced + reversible**, and a `crumb uninstall`/`init --remove-integrations` that strips the managed blocks and registrations cleanly.

A complete, buildable design for this — flag surface, the exact managed-block content, `.mcp.json` merge semantics, the three hook payloads (including the cost-aware `guard` hook), reversal, a `crumb doctor`, and a stdlib-only Python reference sketch matching breadcrumbs' existing conventions — is in **Appendix A**. It is written to be implemented in the breadcrumbs repo, not here.

## 8. Secondary improvement ideas

- **`crumb schema [--json]`** and a **record-template emitter** (`crumb remember attempt --template`) so an agent reads the contract once instead of probing `--help` seven times (§6.2).
- **Named flags for attempt sections** (`--problem`, `--tried`, `--result`, `--do-not-retry`) so the fixed contract is visible in `--help` and unsatisfiable states are unrepresentable.
- **`crumb note question|trap|idea`** (§6.6) — a validate-gated, projection-refreshing writer for the three record kinds that currently have no command, with a matching `memory_note` MCP tool. Closes the read/write asymmetry where `resume`/`audit` surface these but nothing writes them.
- **Summarize the diff-stat in `capture`** (§6.1) — highest-value bug fix after bootstrapping.
- **Surface MCP in `crumb --help`** and/or add a thin `crumb mcp serve` alias for discoverability (§6.3).
- **Identifier/path allowlist in the entropy scanner** (§6.4).
- **A `crumb doctor`** that reports integration health: is the MCP server registered? are hooks installed? is an adapter block present and unbloated? is the resume-packet stale vs HEAD? This gives the agent (and developer) a single "is memory actually wired up?" check.
- **First-run nudge:** if `init` ran but no adapter/MCP/hook integration was chosen, print a one-line reminder that the store will only be used if the agent is told to use it — converting the §5 failure mode into an explicit, visible choice rather than a silent default.

## 10. Closing

crumb-kit's hard parts are already done and done well: a deterministic, safety-conscious, agent-shaped memory engine with a genuinely excellent `guard` and a sophisticated awareness that memory is itself an attack surface. The gap is not capability — it is the last mile of *delivery*. The tool builds a high-quality memory store and then trusts an agent to remember to use it, which this evaluation shows is precisely the thing agents do not do.

Make `crumb init` assemble what the package already contains — inject a managed signpost block into the agent-guidance files, register the MCP server, offer the three hooks — and the system stops depending on agent discipline. That one change is worth more than every other item in this review combined, and the evidence for it is that a capable agent, having built the store itself, walked right past it twice.

## Appendix A — `crumb init` bootstrapper: design concept & reference implementation

> **Status: design concept for the breadcrumbs repo.** Nothing here is wired into a target project automatically by the current 0.1.1; this section specifies the work. Code is illustrative (stdlib-only, matching `cli.py` conventions) and not drop-in. Two external contracts — Claude Code's `.mcp.json` shape and its `settings.json` *hooks* schema — should be re-verified against current Claude Code docs at build time, since they evolve; the field names below reflect the schema as understood at review time.

### A.0 The thesis in one line
`crumb init` already owns every ingredient (the `ADAPTER_FILENAMES` constant, the fenced managed-block writer proven on `.gitignore`, and the `breadcrumbs-mcp` server). The bootstrapper just assembles them, behind explicit consent, reversibly. It turns "a memory store exists" into "the agent is told to use it, given native tools to use it, and reflexively prompted to use it."

### A.1 Principles
1. **Detect, don't assume.** Only touch files/harnesses that already exist in the project. Never create a `CLAUDE.md` where the project doesn't have one unless asked.
2. **Consent + non-interactive parity.** Interactive `init` prompts (as it already does for session-tracking); CI/headless installs use explicit flags. No silent writes to agent-guidance files.
3. **Every edit fenced and idempotent.** Reuse the existing managed-block mechanism so re-running `init` updates in place and never clobbers user content.
4. **Fully reversible.** `crumb init --remove-integrations` strips every managed block and registration the tool added, and nothing else.
5. **The tool owns the translation layer.** Hooks call thin `crumb hook <event>` subcommands so harness-specific stdin/stdout contracts live in breadcrumbs, not in brittle shell one-liners inside `settings.json`.

### A.2 New CLI surface
```
crumb init [existing flags]
           [--with-adapter[=CLAUDE.md,AGENTS.md,...] | --no-adapter]
           [--with-mcp        | --no-mcp]
           [--with-hooks[=session,guard,capture] | --no-hooks]
           [--harness=claude-code|cursor|cline|windsurf|auto]   # default: auto-detect
           [--remove-integrations]                              # reverse everything
           [--print-integrations]                               # dry-run: show what WOULD be written

crumb hook session   # SessionStart: emit resume packet as harness context
crumb hook guard     # PreToolUse:  read proposed action from stdin, emit verdict
crumb hook capture   # Stop/SessionEnd: snapshot a session record

crumb doctor [--json]   # report integration health (see A.7)
crumb note question|trap|idea ...   # §6.6 write-surface fix (separate from bootstrapper)
```
Interactive default (no flags) detects what's present and asks once per integration with a recommended default of **yes** for adapter+mcp and **ask** for hooks (hooks change run-time behavior, so they warrant a deliberate opt-in).

### A.3 Detection
```python
HARNESS_SIGNATURES = {
    "claude-code": ["CLAUDE.md", ".claude/", ".mcp.json"],
    "cursor":      [".cursorrules", ".cursor/"],
    "cline":       [".clinerules"],
    "windsurf":    [".windsurfrules"],
    "copilot":     [".github/copilot-instructions.md"],
}
# Adapter files come from the EXISTING constant; intersect with what's on disk.
present_adapters = [n for n in ADAPTER_FILENAMES if (root / n).exists()]
```
If nothing is detected, `init` still scaffolds `.project-memory/` exactly as today and prints a one-line nudge (A.8) rather than writing anything.

### A.4 Artifact 1 — managed signpost block in adapter files
Reuse the **exact** fenced-block writer already used for `.gitignore` (`_rewrite_managed_block(path, marker, body)`), just pointed at each present adapter file. Content is deliberately tiny (well under `ADAPTER_BLOAT_CHARS = 4000`, so the tool's own `audit` stays green — it practices what it preaches):

```markdown
<!-- >>> breadcrumbs managed block (managed by `crumb init`) — edit above/below, not inside >>> -->
## Project memory (breadcrumbs)

This repo has a durable memory store under `.project-memory/`. Use it:

- **Starting work / new session:** read the resume packet first —
  `crumb resume` (or MCP resource `memory://resume-packet`).
- **Before any risky or irreversible action** (deletes, force-push, schema
  or build-system changes, rewrites): `crumb guard "<action>"` and honor a
  `PAUSE` / `ASK_HUMAN` verdict.
- **After a durable decision or a failed approach:** `crumb remember decision|attempt …`.
- **Leaving a note for the next agent:** `crumb note question|trap|idea …`.
- **Session end:** `crumb capture session`.

Memory must never contain secrets; `crumb scan-secrets` gates commits.
<!-- <<< breadcrumbs managed block <<< -->
```
Placement: append if absent; update-in-place if the markers exist. This is the single highest-value artifact because it lands the usage contract in the file the harness already auto-loads every session — the §5 fix that requires no new harness feature at all.

### A.5 Artifact 2 — MCP server registration (`.mcp.json`)
Merge (don't overwrite) a `breadcrumbs` entry into project `.mcp.json`:
```json
{
  "mcpServers": {
    "breadcrumbs": { "command": "breadcrumbs-mcp", "args": [], "env": {} }
  }
}
```
Merge semantics: parse existing JSON if present; add/replace only the `mcpServers.breadcrumbs` key; preserve all other servers and formatting as much as a stdlib JSON round-trip allows. Guard on the `[mcp]` extra being installed (`mcp_server.sdk_available()`); if not, write the entry anyway but warn that the user must `pip install 'crumb-kit[mcp]'`. For non-Claude harnesses, write the equivalent registration that harness expects (Cursor/Cline have their own MCP config locations — table them per `--harness`).

### A.6 Artifact 3 — harness hooks (the automaticity payoff)
For Claude Code, merge three hooks into `.claude/settings.json` (project-scoped). Each calls a `crumb hook` subcommand so the JSON-contract translation lives in breadcrumbs:

```json
{
  "hooks": {
    "SessionStart": [
      { "hooks": [ { "type": "command", "command": "crumb hook session" } ] }
    ],
    "PreToolUse": [
      { "matcher": "Bash|Edit|Write|MultiEdit",
        "hooks": [ { "type": "command", "command": "crumb hook guard" } ] }
    ],
    "Stop": [
      { "hooks": [ { "type": "command", "command": "crumb hook capture" } ] }
    ]
  }
}
```

**`crumb hook session`** — reads the SessionStart stdin payload, emits the bounded resume packet as additional context (for Claude Code, stdout on SessionStart is added to context; or emit `{"hookSpecificOutput":{"hookEventName":"SessionStart","additionalContext":"<resume packet>"}}`). This auto-loads memory with zero agent action and incidentally fixes the §6.5 staleness-drift-in-practice problem.

**`crumb hook capture`** — on Stop/SessionEnd, runs the equivalent of `capture session --fast` (which **must** first adopt the diff-stat-summary fix from §6.1, or every session end re-introduces the bloat/false-positive). Writes the session record so findings persist even when the agent forgets — the direct fix for the §5.1 "empty search" failure.

**`crumb hook guard` — the one with real design tension.** A PreToolUse hook fires on *every* matched tool call; naively shelling out to `crumb guard` on each is too slow and too noisy. The hook must be cost-aware:
1. Read the PreToolUse stdin (`tool_name`, `tool_input`); derive an action string (Bash → `tool_input.command`; Edit/Write → `file_path` + intent).
2. **Cheap local pre-filter first:** only escalate to full guard scoring when the action matches a risk pattern (the tool already ships risk regexes — e.g. force-push / `--stop` / `clean` / `rm -rf` / `drop table` / schema or migration paths / known-trap file overlaps). Non-risky calls return an immediate silent allow without touching the record store.
3. For risky actions, run guard and map the verdict to the harness contract:
   - `PROCEED` → allow silently.
   - `PAUSE` → **allow but inject the cited records as context** (`permissionDecision: "allow"` + reason), so the agent sees the warning without a hard stop.
   - `ASK_HUMAN` → `permissionDecision: "ask"` with the matched records as the reason, handing the decision to the human.
4. Never *deny* outright from memory alone — memory informs, the human/agent decides. (A `deny` mode could be offered behind an explicit `--strict-guard` opt-in.)

This hybrid keeps the common path nearly free while still firing the §5.1 near-miss *before* the action instead of in a postmortem. The pre-filter pattern set should be shared code with `audit`'s existing risk/instruction-like regexes.

### A.7 Reversal & health
- **`crumb init --remove-integrations`** — strip each managed block by its markers, remove the `mcpServers.breadcrumbs` key, and remove only the hooks whose command is a `crumb hook …`. Leave every surrounding user edit intact. Idempotent.
- **`crumb doctor`** — one-shot integration health for "is memory actually wired up?": adapter block present & unbloated? `.mcp.json` entry present & `[mcp]` extra importable? the three hooks installed & pointing at this `crumb`? resume-packet stale vs HEAD? Returns non-zero if the store exists but no integration is active — the machine-checkable version of the §5 finding.

### A.8 First-run nudge (cheap, high-value)
If `init` completes but the user declined/omitted all integrations, print one line:
```
note: .project-memory/ is set up, but no agent integration was installed.
      The store will only be used if your agent is told to use it.
      Run `crumb init --with-adapter --with-mcp` (and `--with-hooks`) to wire it up,
      or `crumb doctor` to check status later.
```
This converts the silent §5 failure mode into an explicit, visible choice.

### A.9 Reference sketch (illustrative; stdlib-only)
```python
def cmd_init(args):
    root = resolve_root(args)
    scaffold_project_memory(root)            # existing behavior, unchanged
    rewrite_gitignore_managed_block(root)    # existing behavior, unchanged

    plan = IntegrationPlan.detect(root, harness=args.harness)   # A.3
    plan = plan.apply_flags(args)                               # A.2 consent/flags
    if args.print_integrations:
        return plan.print_dry_run()

    for adapter in plan.adapters:            # A.4 — reuse _rewrite_managed_block
        rewrite_managed_block(root / adapter, ADAPTER_MARKER, ADAPTER_BODY)
    if plan.mcp:
        merge_mcp_json(root / ".mcp.json", "breadcrumbs",
                       {"command": "breadcrumbs-mcp", "args": [], "env": {}})  # A.5
    if plan.hooks:
        merge_claude_hooks(root / ".claude/settings.json", plan.hooks)        # A.6

    if not (plan.adapters or plan.mcp or plan.hooks):
        print(FIRST_RUN_NUDGE)               # A.8
    return 0
```

### A.10 Open design questions / risks to resolve in breadcrumbs
- **Hook/`.mcp.json` schema drift.** Claude Code's hook event names, matcher syntax, and the `permissionDecision` contract change over time; pin to a documented schema version and add a `crumb doctor` check that warns on mismatch.
- **PreToolUse cost ceiling.** Even with the pre-filter, validate the per-call overhead is negligible on non-risky actions (target: no record-store I/O on the common path).
- **Multi-harness fan-out.** Cursor/Cline/Windsurf MCP + rules locations differ; ship Claude Code first, table the others behind `--harness`, and keep the managed-block content identical across them.
- **`capture --fast` dependency.** The `Stop` hook is only safe to ship *after* §6.1 (diff-stat summary), or it will re-introduce the bloat/false-positive on every session.
- **Consent durability.** Decide whether `--with-hooks` is remembered in `manifest.yml` so re-running `init` doesn't re-prompt, vs. re-confirming each time.
